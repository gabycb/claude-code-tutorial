Claude Code is Anthropic's AI coding assistant that lives in your terminal and IDE. Unlike autocomplete tools, it operates as a true autonomous agent — it reads your codebase, edits files, runs shell commands, manages git, and self-corrects based on test output.

This tutorial distills the most impactful techniques from Boris Cherny (Claude Code's creator), the Anthropic engineering team, and practitioners who have shared their workflows publicly. Every tip here is battle-tested and sourced.

01

The Right Mental Model

Most users approach Claude Code like a search engine — one-shot question and answer. The Anthropic team uses it like a junior engineer who can read the whole codebase, run tests, and iterate independently.

Three Ways to Run Claude Code

SurfaceBest For
Terminal (CLI)Autonomous long-running tasks, scripting, headless servers
IDE Extension (VS Code / JetBrains)Interactive editing with inline diff previews
Claude Desktop AppMulti-session management, built-in worktree UI

The Most Important Mindset Shift

Boris Cherny, Creator of Claude Code

"Give Claude a way to verify its work. This is the single highest-leverage thing you can do. It gives a 2–3× quality improvement by itself."

The principle: don't review Claude's output yourself — give Claude a feedback loop so it can review its own output. Include test commands, linter checks, or expected outputs directly in your prompts.

Your First Session Checklist

1
Install Claude CodeRun npm install -g @anthropic-ai/claude-code, then claude in any project directory.
2
Create a CLAUDE.md filePlace it in your project root. This is Claude's persistent memory. Start with project overview, coding standards, and key commands.
3
Enable Plan Mode firstType /plan before any complex task. Let Claude draft an approach before writing a single line of code.
4
Give Claude a test commandTell Claude: "After each change, run npm test and fix any failures before moving on."
02

CLAUDE.md — Your Project's Brain

CLAUDE.md is the most powerful configuration file in Claude Code. It persists across sessions, giving Claude long-term project memory without re-explaining context every time.

What to Put In CLAUDE.md

  • Project purpose and architecture — 2–3 sentences max
  • Code style rules — formatting, naming conventions, preferred patterns
  • Key file locations — config files, shared utilities, important modules
  • Build & test commands — exact commands to run tests, lints, builds
  • Off-limits areas — files or patterns Claude should not touch without asking
  • Common gotchas — known quirks, required env vars, external dependencies
Critical Rule — Keep It Under 200 Lines

Anthropic's internal data shows instruction adherence measurably drops once CLAUDE.md exceeds ~200 lines. Link to external docs rather than pasting content inline.

Starter Template

# Project: [Name]
[One paragraph: what this project does and its architecture]

## Code Style
- Language: TypeScript strict mode
- Formatter: Prettier (runs on save)
- Imports: absolute paths from src/

## Key Commands
- Test:  npm test
- Lint:  npm run lint --fix
- Build: npm run build

## Important Files
- Config:       src/config/index.ts
- Shared utils: src/lib/

## Rules
- Never modify migration files directly
- Always add tests for new features
- Run the full test suite before marking a task done
03

Plan Mode — Think Before You Code

The biggest mistake: letting Claude jump straight into coding without a plan. Plan Mode separates exploration from execution, preventing Claude from solving the wrong problem.

1
Enter Plan ModeType /plan at the start of any complex task. Claude researches your codebase and proposes an approach — no code written yet.
2
Review the plan criticallyAsk clarifying questions. Push back on assumptions. The Anthropic team has a second Claude session review the plan "as a staff engineer" before approving.
3
Approve and implementOnce the plan looks right, approve it. Claude exits plan mode and begins implementation following the agreed approach.
Pro Tip — Two-Claude Plan Review

Boris Cherny's team runs two Claude sessions: one drafts the plan, a second reviews it from a fresh context as a "staff engineer." Fresh context means no bias toward the approach just suggested — it catches blind spots the first Claude misses.

When to Use Plan Mode

Use Plan Mode When…Skip It When…
Adding a major new featureFixing a small, clear bug
Refactoring across multiple filesWriting a simple utility function
You're unsure of the right approachThe implementation path is obvious
The task has many dependenciesYou need a quick one-liner
04

Parallel Sessions & Git Worktrees

This is the #1 productivity tip from the Claude Code team. Running multiple Claude sessions simultaneously — each in its own git worktree — is how Anthropic engineers build at speed.

Boris Cherny, on Threads

"Spin up 3–5 git worktrees at once, each running its own Claude session in parallel. It's the single biggest productivity unlock. I have dozens of Claudes running at all times, and this is how I do it."

What Are Git Worktrees?

A git worktree lets you check out a separate branch in a different directory while sharing the same .git folder. Unlike cloning the whole repo, worktrees are lightweight — each has its own working files, staged changes, and Claude session.

Setting Up Parallel Sessions

# Start a session in its own worktree (auto-named)
claude --worktree

# Give the worktree a descriptive name
claude --worktree feature-auth

# Add quick shell aliases for navigation
alias wta='cd ~/code/project-worktree-a'
alias wtb='cd ~/code/project-worktree-b'

# Clean up after merging
git worktree remove ../feature-auth

Practical Workflow

  • One worktree per feature or bug — sessions never clobber each other
  • Dedicated "analysis" worktree — read-only investigation (logs, queries) isolated from coding sessions
  • Fresh-context code review — a worktree reviewer has no bias toward code it just wrote
  • Claude Desktop app — check the "worktree" checkbox when starting a new session for a GUI workflow
05

Context Management — Your #1 Failure Mode

Context degradation is the primary reason Claude Code output gets worse over a long session. As the context fills with failed attempts and discarded ideas, quality drifts.

Warning Signs of Context Pollution

  • Claude repeats the same fix with slight variations
  • Responses reference earlier failed approaches
  • Claude loses track of which file to edit
  • Suggestions contradict earlier decisions in the session

Techniques

Technique 1 — Push Exploration to a Subagent

Use a dedicated subagent or separate worktree for research. Only the final summary comes back to your main session — not all the dead ends.

Technique 2 — Use /compact Proactively

Don't wait for auto-compact. Run /compact when a task is done and before starting the next. Add a /compact policy to CLAUDE.md so Claude knows what to preserve.

Technique 3 — Fresh Sessions for Code Review

Never have the same session that wrote the code also review it. A fresh context is structurally forced to be objective.

Technique 4 — Explicit Reset

Tell Claude: "Ignore the approach we just tried. Start fresh from the original file." This resets Claude's working hypothesis without starting a whole new session.

06

Subagents — Specialized AI Teammates

Subagents are specialized Claude instances defined in .claude/agents/. Each runs in its own isolated context window, has its own tool permissions, and handles a specific bounded task. The main session orchestrates; subagents execute.

Main Agent RoleSubagent Role
Plans the overall approachExecutes one specific task
Owns integration and mergingHas isolated, clean context
Reads subagent summariesOnly results come back — not exploration
Decides what to delegateHas scoped tool access

Creating a Subagent

# File: .claude/agents/test-runner.md
---
name:        test-runner
description: Runs the test suite and reports failures with root cause analysis
tools:       [Bash, Read, Grep]
---

You are a specialized test runner agent.
When invoked, run the full test suite: npm test

For each failure:
1. Identify the failing test and the assertion
2. Locate the relevant source file
3. Provide a concise root cause (1–2 sentences)

Return a structured list of failures with root causes.
Do not fix anything — report only.

Common Subagent Patterns

  • code-reviewer — reviews a PR diff for bugs; has no knowledge of why the code was written
  • test-runner — executes tests and returns structured failure reports
  • security-scanner — checks for OWASP vulnerabilities in new code
  • frontend-qa — takes a screenshot and checks UI against the design spec
  • db-analyst — reads logs and queries production data (read-only tools only)
07

Hooks — Deterministic Guardrails

Hooks are shell scripts that fire automatically on specific Claude Code events. Unlike CLAUDE.md instructions (which Claude can misinterpret), hooks execute deterministic code — they cannot hallucinate. Rules enforced by hooks are system-level guarantees.

Hook EventWhen It Fires
PreToolUseBefore Claude calls any tool — can block or modify the call
PostToolUseAfter a tool call completes — inspect results, run follow-ups
StopWhen a session ends — run cleanup, send notifications
SubagentStopWhen a subagent finishes — collect results

Practical Examples

Auto-run Tests After File Edits

PostToolUse hook on Write/Edit: automatically run npm test whenever Claude edits a source file. Claude sees the output and fixes failures without you intervening.

Block Protected Files

PreToolUse hook: if Claude tries to write to migrations/, schema.prisma, or .env, block the call and return an error. Hard enforcement, not a prompt instruction.

Slack Notification on Session End

Stop hook: post a Slack message summarizing what Claude did when a long-running autonomous session completes. No need to watch the terminal.

Hook Config in settings.json

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type":    "command",
        "command": "npm test 2>&1 | tail -20"
      }]
    }],
    "Stop": [{
      "hooks": [{
        "type":    "command",
        "command": "scripts/notify-slack.sh"
      }]
    }]
  }
}
08

Prompting Techniques for Better Output

The Evidence Rule — From Anthropic's Internal Playbook

"Have Claude show evidence rather than assert success. Require the test output, the command it ran, and the result — not just 'it works.' Reviewing evidence is faster than re-running verification yourself."

Weak vs. Strong Prompts

Weak

"Add tests"

Strong

"Add unit tests for createUser() in src/users/service.ts. Run npm test after and fix any failures. Show me the test output."

Weak

"Fix this bug"

Strong

"The login endpoint returns 500 when the email contains a + character. Reproduce it first, then fix it. Don't change the email validation regex — that's tested separately."

Weak

"Make it faster"

Strong

"getUserFeed() takes 800ms in production (check logs/slow-queries.log). Profile it, identify the bottleneck, and propose a fix before implementing."

Weak

"Clean up this file"

Strong

"Refactor src/auth/middleware.ts for readability. Don't change behavior. Run the full test suite after to confirm nothing broke."

Self-Verification Phrases

  • "After implementing, run the tests and paste the output here"
  • "Before you finish, check that the TypeScript compiler has no errors"
  • "Show me which files you changed and why"
  • "If any test fails, fix it before reporting done"

Onboarding Claude to a New Codebase

  • "Read the README and src/index.ts, then summarize what this service does"
  • "Find all the places we make HTTP calls to external services"
  • "What's the data model for users? Trace it from the database to the API"

Dynamic Workflows — Fleet-Scale Automation

Type the word "workflow" in a prompt and Claude writes an orchestration script on the fly, then spins up a coordinated fleet of subagents to tackle your most complex tasks in parallel.

Introduced by Anthropic — Research Preview

Dynamic workflows can spawn up to 1,000 subagents working in parallel. Claude writes the coordination logic itself — you describe the goal and constraints; Claude figures out the parallelization strategy.

Example Workflow Prompts

# Large-scale migration
"Create a workflow to migrate all our REST endpoints to our new
error-handling pattern. There are 47 endpoints across 12 files."

# Monorepo security scan
"Create a workflow to run security scanning across all services
in the monorepo simultaneously and produce a consolidated report."

# TypeScript upgrade
"Create a workflow to update all TypeScript files from v4 patterns
to v5, running tests after each batch of changes."

When to Use Dynamic Workflows

  • Repetitive changes across many files (migrations, refactors, updates)
  • Tasks that can be cleanly parallelized with independent subtasks
  • Large-scale audits or analysis across an entire codebase
  • Any task where the pattern is clear but the volume is high

MCP Servers & Tool Integrations

Model Context Protocol (MCP) extends Claude Code with custom tools — databases, APIs, internal services, observability platforms. Each MCP server gives Claude new capabilities beyond its built-in toolset.

MCP ServerWhat It Enables
GitHub MCPCreate PRs, manage issues, review code directly in Claude sessions
Postgres / MySQL MCPQuery your database — Claude reads schema and writes queries with context
Slack MCPPost summaries, get notifications, pull channel context into sessions
Sentry / DataDog MCPClaude reads error logs and traces to diagnose production issues
Linear / Jira MCPClaude reads tickets and updates status as it implements features
The Highest-Impact Plugin — LSP

Install the LSP (Language Server Protocol) plugin. It gives Claude automatic diagnostics after every file edit — type errors, unused imports, missing return types — without you running a separate check. Described by the Anthropic team as the single highest-impact plugin you can install.

Advanced Patterns from Power Users

The Layered Configuration System

LayerPurpose
CLAUDE.md — RulesDurable project policy: what to do, what to avoid, how to communicate
.claude/agents/ — ProceduresSpecialized skills for bounded, repeatable tasks
settings.json Hooks — ChecksDeterministic enforcement: system-level guardrails
Git Worktrees — IsolationContext isolation for parallel and independent work

The Analysis Worktree Pattern

Keep a permanently open "analysis" worktree dedicated to read-only investigation — reading logs, querying databases, exploring production data. This keeps analytical context completely separate from implementation sessions.

The Fresh-Eyes Review Pattern

Never have the same Claude session that wrote the code also review it. Open a new worktree, paste only the diff, and ask for a review. The reviewer is structurally forced to be objective — it has no context about why decisions were made.

The Hypothesis Isolation Pattern

From Advanced Practitioners

When debugging a hard problem, spawn a new Claude session for each hypothesis you want to test. Each starts fresh with your hypothesis as the only context. This prevents earlier failed theories from polluting later reasoning.

The 10 Rules to Code By

  • Give Claude a verification loop — always include test commands in your prompts
  • Plan before coding — use /plan on any non-trivial task
  • Keep CLAUDE.md under 200 lines — quality over quantity
  • Run 3–5 parallel worktree sessions — the team's biggest productivity unlock
  • Use subagents for bounded tasks — preserve main context for orchestration
  • Enforce rules with hooks, not just prompts — hooks cannot hallucinate
  • Fresh context for code review — never review what you just wrote
  • Show evidence, not assertions — require proof of success
  • Separate analysis from implementation — dedicated investigation worktree
  • Ask Claude to prompt itself — self-correction loops beat manual iteration
