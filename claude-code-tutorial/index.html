Most developers use Claude Code like a smarter autocomplete — typing a prompt, getting code, copy-pasting it in. That's leaving most of the value on the table.

The Anthropic team treats Claude Code as a junior engineer on the team: one who can read the whole codebase, run tests, manage git, and iterate independently when given the right feedback loops and project context. This tutorial shows you exactly how to set that up.

Who This Is For

Developers who have installed Claude Code and used it at least a little, but want to get dramatically more out of it. Not a beginner's installation guide — a practitioner's playbook.

Part 1 — Foundations
01 Foundation

The Verification Loop

This is the single highest-leverage technique in all of Claude Code. Everything else in this tutorial builds on it. Do this one thing and you will immediately see 2–3× better results.

Boris Cherny, Creator of Claude Code — verified on X

"Probably the most important thing to get great results out of Claude Code — give Claude a way to verify its work. If Claude has that feedback loop, it will 2–3x the quality of the final result."

The Core Idea

Don't review Claude's output yourself. Give Claude a way to see the results of its own work — test output, compiler errors, lint warnings, a browser screenshot — so it can self-correct before reporting back to you.

Before & After

❌ Without a Loop

"Add input validation to the signup form."

→ Claude writes code, says "Done!"
→ You run it. Runtime error.
→ You paste the error back.
→ Back and forth. You do the work.

✅ With a Loop

"Add input validation to the signup form. After each change run npm test, fix any failures, and show me the final test output before reporting done."

→ Claude writes, tests, catches the error, fixes it, and comes back with proof.

Verification by Task Type

Boris's own breakdown of what verification looks like across different task types:

Task TypeVerification Method
Backend / APIClaude starts the server and tests it end-to-end with real HTTP requests
Frontend / UIUse the Claude Chromium extension so Claude controls a real browser and checks visual output
Desktop appsUse computer use to let Claude interact with the running app
Any codeRun the test suite + linter as a PostToolUse hook so it fires automatically after every file edit

The One-Line Rule

Add this to every non-trivial prompt: "Run [test command] after each change. Fix any failures before reporting done. Show me the final output."

Or automate it with a hook (see Section 07) so you never have to type it.

02 Foundation

CLAUDE.md — Project Memory

CLAUDE.md is a file in your project root that Claude Code reads at the start of every session. It gives Claude persistent project memory — so you don't re-explain your stack, conventions, and key commands every time.

The <200 Line Rule

Anthropic's internal data shows that instruction adherence measurably drops once CLAUDE.md exceeds ~200 lines. Keep it concise. Link to external docs rather than pasting content inline.

What Goes In

  • Project purpose and architecture — 2–3 sentences. What does this repo do?
  • Build, test, and lint commands — exact commands, not descriptions
  • Code style rules — only what a linter won't catch (naming, patterns, conventions)
  • Key file locations — config files, shared utilities, entry points
  • Off-limits areas — files Claude should never touch without explicit approval
  • Known gotchas — env vars needed, external services, quirks of the codebase

What Stays Out

  • Generic advice Claude already knows ("write readable code", "handle errors")
  • Everything that can be auto-discovered by reading the codebase
  • Procedures that change often — keep those in subagent definitions instead

Starter Template

# Project: [Name]
# [2-3 sentences: what this repo does and its high-level architecture]

## Commands
- Build: npm run build
- Test:  npm test
- Lint:  npm run lint --fix
- Single test: npx jest path/to/file.test.ts

## Code Style
- TypeScript strict mode
- Absolute imports from src/
- No default exports

## Key Files
- Config:  src/config/index.ts
- Utilities: src/lib/
- API routes: src/routes/

## Rules
- Never edit migration files directly
- All new API routes need integration tests
- Run the full test suite before marking any task done
03 Foundation

Plan Mode

The biggest mistake most developers make: letting Claude jump straight into coding without a plan. Plan Mode forces a separation between understanding the problem and writing the solution — preventing Claude from confidently solving the wrong thing.

How to Use It

1
Enter Plan ModeType /plan before describing any complex task. Claude will research your codebase and propose an approach without writing any code.
2
Review criticallyPush back on assumptions. Ask "what files did you look at?" and "what alternatives did you consider?" The plan should feel specific to your codebase, not generic.
3
Approve and implementOnce the plan is right, approve it. Claude exits plan mode and implements the agreed approach.
Pro Technique — Two-Claude Plan Review

The Anthropic team runs two Claude sessions: one drafts the plan, a second reviews it cold — as if it just walked into the room with no context. Fresh context = no bias toward the approach just proposed. This catches blind spots the first Claude misses.

When to Use Plan Mode

Skip It

Fixing a small clear bug · Adding a one-liner · Tasks with an obvious implementation path

Use It

Any new feature · Refactoring across multiple files · Anything touching shared infrastructure · When you're unsure of the right approach

Part 2 — Power Techniques
04 Power Technique

Parallel Sessions & Git Worktrees

Running 3–5 Claude sessions simultaneously — each in its own git worktree — is the #1 productivity tip from the Claude Code team. Most developers use one session at a time and leave enormous throughput on the table.

Boris Cherny — verified on X

"Spin up 3–5 git worktrees at once, each running its own Claude session in parallel. It's the single biggest productivity unlock, and the top tip from the team. I have dozens of Claudes running at all times, and this is how I do it."

Why Worktrees, Not Clones

A git worktree lets you check out a separate branch in a different directory while sharing the same .git folder. Unlike cloning the whole repo, worktrees are lightweight — each has independent working files and staged changes, but shares history and objects. No disk bloat, no config duplication.

Setting Up

# Start Claude in its own auto-named worktree
claude --worktree

# Name it after the task
claude --worktree feature-auth

# Or in Claude Desktop: check "worktree" when starting a session

# Shell aliases for instant navigation between sessions
alias wta='cd ~/code/project-worktree-a'
alias wtb='cd ~/code/project-worktree-b'

# Clean up after merging
git worktree remove ../feature-auth

Recommended Session Layout

SessionPurposeTools Allowed
Session A–CActive feature or bug work — one per taskFull
Analysis worktreePermanent read-only session for log reading, DB queries, production investigationRead-only
Review worktreeFresh-context code review — starts cold, sees only the diffRead + Grep
05 Power Technique

Context Management

Context degradation is the #1 failure mode for long Claude Code sessions. As failed attempts, debugging tangents, and discarded approaches accumulate, output quality drifts. Managing context is an active skill, not a passive one.

Warning Signs

  • Claude repeats the same fix with minor variations
  • Responses reference earlier failed approaches as if they might still work
  • Claude loses track of which file is canonical
  • Suggestions contradict decisions made earlier in the session

Techniques

Technique 1 — Push Exploration to a Subagent

Use a dedicated subagent or a separate worktree session for research and investigation. Only the final summary returns to your main session — not all the dead ends that polluted the search.

Technique 2 — /compact Proactively

Don't wait for auto-compact. Run /compact when a task is done and before starting the next. Add a /compact policy to CLAUDE.md so Claude knows what to preserve during compression.

Technique 3 — Explicit Hypothesis Reset

When Claude is stuck in a loop, say: "Ignore the approach we just tried. Start fresh from the original file as it exists on disk right now." This resets Claude's working theory without a full session restart.

Technique 4 — Fresh Session for Review

Never have the same session that wrote the code also review it. Start a new worktree, pass only the diff, and ask for review. The reviewer is structurally forced to be objective — it has no memory of why choices were made.

06 Power Technique

Subagents

Subagents are specialized Claude instances defined in .claude/agents/. Each runs in its own isolated context window with its own tool permissions. The main session orchestrates; subagents execute bounded tasks cleanly.

Main SessionSubagent
Plans the overall approachExecutes one specific bounded task
Owns integration and mergingHas clean isolated context
Reads the summaryDead ends stay contained — don't pollute main session
Decides what to delegateHas scoped tool access — can't accidentally do more

Defining a Subagent

# .claude/agents/test-runner.md
---
name:        test-runner
description: Runs the test suite, reports failures with root cause analysis
tools:       [Bash, Read, Grep]
---

You are a specialized test runner. When invoked:
1. Run npm test
2. For each failure: find the source file, identify the assertion, give a 1-2 sentence root cause
3. Return a structured list of failures with root causes

Do not fix anything. Report only. Keep summaries under 3 lines per failure.

Useful Subagent Patterns

  • code-reviewer — reviews a diff cold; never knows why the code was written
  • test-runner — runs tests, returns structured failure reports with root causes
  • security-scanner — checks new code for OWASP top 10; read-only tools only
  • db-analyst — queries production data; strictly read-only tool set enforced via agent definition
  • frontend-qa — takes a screenshot via browser extension, checks against design spec
07 Power Technique

Hooks

Hooks are shell scripts that fire automatically on Claude Code events. Unlike instructions in CLAUDE.md (which Claude can misinterpret), hooks execute deterministic code. Hooks cannot hallucinate. They enforce your rules at the system level.

EventWhen It FiresCommon Uses
PreToolUseBefore any tool call — can block itBlock writes to protected files; enforce branch naming
PostToolUseAfter a tool call completesAuto-run tests; lint after every file edit
StopWhen the session endsSlack notification; write session summary to a log
SubagentStopWhen a subagent finishesCollect and validate subagent outputs

The Auto-Test Hook (Most Impactful)

This hook fires after every file edit and feeds the test output back to Claude. Combined with Section 01, Claude will never need you to manually re-run tests.

// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type":    "command",
        "command": "npm test 2>&1 | tail -30"
      }]
    }],
    "Stop": [{
      "hooks": [{
        "type":    "command",
        "command": "scripts/notify-slack.sh \"Session complete\""
      }]
    }]
  }
}

Protecting Files with PreToolUse

// Block writes to migrations/ and .env
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type":    "command",
        "command": "scripts/check-protected-paths.sh"
      }]
    }]
  }
}

08 Going Further

Prompting Patterns

The Evidence Rule

Always require proof, not assertion. Instead of accepting "it works," require: "show me the test output, the command you ran, and the result." Reviewing evidence is 10× faster than re-running verification yourself.

Weak vs. Strong Prompts

❌ Weak

"Add tests"

✅ Strong

"Add unit tests for createUser() in src/users/service.ts. Run npm test after and fix any failures. Show me the final test output."

❌ Weak

"Fix this bug"

✅ Strong

"Login returns 500 when the email contains +. Reproduce it first with a curl command, then fix it. Don't change the email validation regex — that has separate tests."

❌ Weak

"Make it faster"

✅ Strong

"getUserFeed() takes 800ms in production. Check logs/slow-queries.log, profile the query, identify the bottleneck, and propose a fix before implementing it."

Onboarding Claude to an Unfamiliar Codebase

Treat Claude like a senior engineer joining your team. Ask the questions you'd ask in onboarding:

  • "Read the README and src/index.ts, then summarize what this service does in 3 sentences"
  • "Find all the places we make HTTP calls to external services and list them"
  • "Trace the data model for a User from the database through to the API response"
  • "What are the main entry points for this codebase?"

Self-Verification Phrases to Add to Any Prompt

  • "Run the tests after implementing and fix any failures before reporting done"
  • "Before you finish, confirm TypeScript compiles without errors"
  • "Show me which files you changed and one sentence on why each change was needed"
  • "If something unexpected comes up, stop and ask me rather than making assumptions"
09 Going Further

Model Selection

Claude Code supports Haiku, Sonnet, and Opus. Choosing the right one for the task matters — the difference in speed and cost is large, but so is the capability gap for hard problems.

Claude Haiku
Fastest · Cheapest
  • Bulk repetitive tasks
  • Fast iteration on simple changes
  • Subagent workers in large workflows
  • Log parsing and data extraction
  • Anything where speed > depth
Claude Sonnet
Balanced · Daily Driver
  • Most day-to-day coding tasks
  • Feature implementation
  • Code review and refactoring
  • Writing tests
  • The right default for 80% of work
Claude Opus
Most Capable · Slowest
  • Hard architectural decisions
  • Debugging subtle multi-file issues
  • Complex plan drafting and review
  • When Sonnet gets stuck
  • Security and performance analysis

Switching Models

# Per session via CLI flag
claude --model opus
claude --model haiku

# In /config for the session default
/config

# In settings.json for a persistent project default
{ "model": "claude-sonnet-4-6" }
Cost Tip

Use Haiku for the "worker" subagents in a dynamic workflow and Opus for the orchestrator. You get Opus-level planning with Haiku-level execution costs on the bulk of the work.

10 Going Further

Dynamic Workflows

Type the word "workflow" in a prompt and Claude writes an orchestration script on the fly, then spins up a coordinated fleet of subagents to execute it in parallel. This is Claude Code's highest-throughput mode.

Anthropic Research Preview

Dynamic workflows can coordinate up to 1,000 subagents in parallel. Claude writes the parallelization strategy itself — you describe the goal and constraints.

Example Triggers

# Large-scale migration
"Create a workflow to migrate all REST endpoints to our new
error-handling pattern. There are 47 endpoints across 12 files."

# Monorepo security audit
"Create a workflow to run security scanning across all services
in the monorepo simultaneously and produce a consolidated report."

# TypeScript upgrade
"Create a workflow to update all files from TypeScript v4 patterns
to v5, running tests after each batch of changes."

When to Use Workflows

  • High-volume repetition — the same transformation across many files
  • Independent subtasks — work that can be parallelized without merge conflicts
  • Large-scale audits — scan an entire codebase for a pattern or issue
  • Not for — tasks with deep interdependencies or where each step depends on the last
11 Going Further

MCP & Integrations

Model Context Protocol (MCP) lets you extend Claude Code with custom tools — databases, APIs, internal services, observability platforms. Each MCP server you connect gives Claude new capabilities beyond its built-in toolset.

MCP ServerWhat It Enables
GitHub MCPCreate PRs, manage issues, trigger workflows — without leaving Claude
Postgres / MySQL MCPClaude reads your schema and writes queries with full codebase context
Slack MCPPost session summaries, pull channel context into conversations
Sentry / Datadog MCPClaude reads real error traces and stack traces to diagnose production issues
Linear / Jira MCPClaude reads tickets as it implements, updates status when done
Highest-Impact Plugin — LSP

Install the Language Server Protocol (LSP) plugin. After every file edit, Claude automatically receives diagnostics — type errors, unused imports, missing return types — without you running a separate check. The Anthropic team describes this as the single highest-impact plugin available.

Configuring MCP Servers

// .claude/settings.json — add MCP servers here
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "<token>" }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres", "<connection-string>"]
    }
  }
}
12 Going Further

Team Workflows

All the techniques above work for solo developers. Here's how to extend them to a team of multiple humans working with Claude simultaneously.

Shared CLAUDE.md Governance

  • Treat CLAUDE.md like code — changes go through PR review, not direct pushes to main
  • One owner per section — assign ownership so rules don't conflict or contradict
  • Separate project CLAUDE.md from personal preferences — personal tweaks go in ~/.claude/CLAUDE.md (global), project rules go in the repo
  • Version the agent definitions too — .claude/agents/ should be committed and reviewed like any other config

Preventing Worktree Conflicts

  • Name worktrees after the ticket or PR — claude --worktree PROJ-1234 makes it clear who owns what
  • One human, one worktree — two people's Claude sessions should never be in the same worktree simultaneously
  • Merge frequently — long-lived worktrees accumulate conflicts; treat them like feature branches

Using Claude for Code Review on a Team

1
Author opens a PR as normalClaude in the author's session should not review its own work.
2
Reviewer starts a fresh worktreeNo context about the PR except the diff. claude --worktree review-pr-1234
3
Pass only the diff"Here is a diff. Review it for correctness bugs and unintended behavior changes. Be direct. Do not comment on style."
4
Human reviews Claude's commentsClaude's review is a first pass, not the final word. A human still approves before merging.
Why Fresh Context Matters for Review

A Claude session that wrote the code will unconsciously defend its own choices. A fresh-context reviewer has no memory of why decisions were made — it evaluates the code on its face. This structural isolation is what makes the review genuinely independent.

Shared Subagent Library

Commit your .claude/agents/ directory to the repo so every team member gets the same specialized agents. Standard agents to share across a team:

  • test-runner — consistent failure reporting format for everyone
  • pr-reviewer — shared review criteria applied uniformly
  • release-notes — generates changelogs from commit history in your project's tone

The 10 Rules

  1. Give Claude a verification loop — always include a test command. This single change gives 2–3× better results.
  2. Plan before coding — use /plan on any non-trivial task. Solving the wrong problem is expensive.
  3. Keep CLAUDE.md under 200 lines — adherence drops past this threshold. Quality over quantity.
  4. Run 3–5 parallel worktree sessions — the team's #1 productivity tip. Don't work serially.
  5. Isolate exploration from implementation — push research to a subagent or separate worktree.
  6. Enforce rules with hooks, not prompts — hooks cannot hallucinate. Prompts can be misread.
  7. Use fresh context for code review — never review what you just wrote. Structural bias is real.
  8. Require evidence, not assertions — "show me the output" beats "trust me, it works."
  9. Match model to task — Haiku for bulk work, Sonnet for daily coding, Opus for hard problems.
  10. Commit your .claude/ directory — agent definitions and hooks are team infrastructure.
